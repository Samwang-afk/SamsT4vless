# CDN、面板证书与 XHTTP 入站设计

## 目标

使用已接入 Cloudflare 的独立域名，为 3x-ui 配置有效 TLS 证书，并新增不影响旧服务的 VLESS + XHTTP 入站。普通访客访问时看到一个虚构的日本地方文化资料站；Windows 单文件客户端最终切换到新链路。

## 边界

- 保留现有 443 REALITY、8388 ss-server、3x-ui 数据和防火墙保护。
- 新服务使用 Cloudflare 支持的 HTTPS 端口 8443。
- 不复制或冒充真实机构，不使用第三方商标、图片或内容。
- 不删除旧入站；新链路验证成功前不覆盖现有客户端 EXE。

## 证书与面板

- 使用现有 acme.sh 通过 HTTP-01 申请公开 CA 证书。
- 私钥仅保存在 VPS，权限限制为 root 可读。
- 先将证书配置到 3x-ui 并重启验证；失败时恢复原面板证书路径。
- 面板继续使用原随机路径和账户认证，原始管理端口不向全网开放。
- Nginx 仅在既有随机面板路径转发至本机 3x-ui，其余请求不会暴露面板。

## HTTPS 前端与伪装站

- 安装 Debian 官方 Nginx，监听 8443 并使用新证书。
- 默认站点为轻量静态日文页面，主题为虚构的地方文化资料室。
- 页面仅包含本地 HTML/CSS，不加载外部统计、字体、图片或脚本。
- 未匹配隐藏路径的请求统一返回正常网站页面或标准 404，不泄露 Xray、3x-ui 或代理信息。

## XHTTP 入站

- 通过 3x-ui API 创建独立 VLESS 用户和 XHTTP 入站。
- Xray 仅监听回环地址上的随机高位端口，不直接暴露公网。
- 使用随机隐藏 XHTTP 路径、`auto` 模式和基础 padding，不依赖自定义请求头。
- Nginx 只将该隐藏路径转发到 Xray，并关闭响应缓存、允许流式请求。
- Cloudflare 至 Nginx 使用 Full (strict) TLS；Nginx 至本机 Xray 使用明文回环 HTTP。

## 客户端

- 复用现有 Mihomo、Wintun、GUI 和流量统计，不增加协议选择界面。
- 节点使用域名、8443、TLS、XHTTP、独立 UUID 和隐藏路径。
- 新构建使用单独文件名；端到端测试通过后才作为分享版本。

## 公开仓库脱敏

- 同步执行已批准的 README 与仓库脱敏设计。
- README 仅介绍项目、构建和示例配置。
- 真实基础设施信息从公开源码、脚本、内部文档和全部可达 Git 历史中移除。
- 内部设计文档保留在本地但不再公开跟踪。

## 验证与回滚

1. 验证域名返回 Cloudflare 地址，源站证书链和主机名有效。
2. 验证面板可登录，Nginx 默认页面不暴露服务信息。
3. 验证 XHTTP 客户端可访问外网且出口为 VPS。
4. 验证 443 REALITY、8388 ss-server、面板和 nftables 重启后仍正常。
5. 运行 Rust 工作区测试，并扫描公开仓库全部可达提交。
6. 任一步失败时先停用本次新增 Nginx/XHTTP 配置；不修改旧入站和旧客户端。
